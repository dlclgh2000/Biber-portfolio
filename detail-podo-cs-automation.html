PODO CS 운영 자동화

PODO는 AI 기반 영어 학습 플랫폼으로, 사용자 급증에 따라 기존 카카오톡 중심의 CS 운영이 한계에 도달했습니다. 모든 문의가 카카오톡 채널로 집중되어 평균 응답 시간이 2시간 이상 소요되었고, 전체 문의의 85%가 환불, 결제 변경, 홀딩 등 반복적인 내용이었습니다. 이 프로젝트는 인앱 고객센터 자동화 시스템을 구축하여 사용자가 즉시 문제를 해결하고, 운영팀은 더 중요한 업무에 집중할 수 있도록 설계되었습니다.

카카오톡으로는 한계가 있었다

사용자가 급증하면서 기존 카카오톡 채널로 하루 평균 100건 이상의 문의가 쏟아졌습니다. 운영팀은 실시간 대응이 불가능했고, 사용자는 최소 2시간 이상 기다려야 했죠. 하지만 데이터를 분석해 보니 놀라운 사실을 발견했습니다.

전체 문의의 85%가 환불, 결제 변경, 홀딩 등 표준화된 프로세스였고, 이는 모두 자동화할 수 있었습니다.

"사용자는 즉시 답을 원하는데, 왜 기다리게 할까?"
이 질문에서 출발하여, 인앱 고객센터 자동화 시스템을 구축했습니다.

즉각적인 해결, 직관적인 경험

1. 검색 우선 인터페이스: 사용자의 78%가 명확한 질문을 가지고 진입합니다. 검색 바를 최상단에 배치하고, 인기 검색어 태그로 원클릭 접근을 가능하게 했습니다. 검색이 카테고리 탐색보다 평균 1.5단계 빠르고, 셀프 해결률이 23% 더 높았습니다.

2. 자동 처리 플로우: 환불, 결제 변경, 수강 홀딩 등 반복적인 요청은 사용자가 직접 처리할 수 있는 셀프 플로우를 구축했습니다. 각 단계마다 현재 상태, 다음 동작, 예상 결과를 명확하게 표시하여 완료율을 85%에서 95%로 높였습니다.

3. FAQ 시스템: 자주 묻는 질문을 카테고리별로 정리하고, 아코디언 UI로 같은 화면에서 여러 질문을 탐색할 수 있게 했습니다. 답변마다 "도움이 되었나요?" 피드백을 수집하여 콘텐츠를 지속적으로 개선했습니다.

4. 대안 제시: 검색 결과가 없을 때도 추천 FAQ를 보여주어 이탈률을 45%에서 18%로 감소시켰습니다. 해결되지 않으면 즉시 1:1 문의로 연결되도록 설계했습니다.

사용자 여정을 3가지 시나리오로 설계

6개월간 5,000건 이상의 CS 데이터를 분석하고, 카드 소팅을 통해 사용자가 직관적으로 이해하는 정보 구조를 도출했습니다. 최초 9개의 카테고리를 5개로 통합하여 인지 부담을 줄였으며, 사용자 여정을 3가지로 정의했습니다.

1. 빠른 검색형: 검색 바 → 자동완성 → 답변 (2단계)
2. 탐색형: 카테고리 선택 → 질문 목록 → 답변 (3단계)
3. 즉시 처리형: 검색/카테고리 → 자동 처리 플로우 시작 (2-4단계)

Figma에서 인터랙티브 프로토타입을 제작하여 실제 사용자 10명을 대상으로 유저빌리티 테스트를 진행했습니다. 검색 바를 상단에 배치했을 때 발견율이 40% 증가했고, 인기 검색어 태그의 클릭률이 일반 카테고리 대비 2.3배 높았습니다.

데이터가 증명한 디자인 결정

1. 아코디언 UI vs 별도 페이지
FAQ를 탐색할 때 사용자는 여러 질문을 빠르게 스캔하는 패턴을 보였습니다(평균 3.2개 확인). 페이지 이동 없이 같은 화면에서 여러 FAQ를 열어보는 아코디언 방식이 효율적이었습니다. 반면 환불이나 결제 변경 같은 중요한 액션은 별도 페이지로 분리했을 때 오류율이 12%에서 3%로 감소했습니다.

2. 단계별 명확성이 완료율을 높인다
초기 프로토타입에서는 간결함을 위해 최소한의 안내만 제공했으나, 사용자의 40%가 "다음에 무슨 일이 일어날지 모르겠다"는 불안감을 표현했습니다. 각 단계별로 프로그레스 바, 현재 단계 설명, 예상 소요 시간을 추가한 결과, 플로우 완료율이 85%에서 95%로 증가했습니다.

3. 검색 결과 없음도 기회다
"검색 결과가 없습니다" 메시지만 표시했을 때 45%의 사용자가 바로 이탈했습니다. 대신 유사한 FAQ 또는 인기 FAQ를 추천했을 때 이탈률이 18%로 감소했고, 이 중 62%가 추천 FAQ를 통해 문제를 해결했습니다.

즉각적인 해결이 만든 변화

출시 후 3개월간 카카오톡 채널로 유입되는 문의가 월평균 2,100건에서 690건으로 67% 감소했습니다. 사용자의 82%가 고객센터 내에서 스스로 문제를 해결했으며, 평균 해결 시간은 2시간에서 3분으로 대폭 단축되었습니다.

"환불 신청이 이렇게 쉬울 줄 몰랐어요. 앱에서 바로 처리되니까 너무 편해요!" 같은 리뷰가 쏟아졌고, 앱스토어 고객센터 관련 긍정 리뷰가 340% 증가했습니다. NPS(Net Promoter Score)는 52점에서 68점으로 상승했습니다.

비즈니스 임팩트도 명확했습니다. CS 담당 인력 2명을 절감하여 연간 약 1억 원의 비용을 절감했고, 운영팀은 단순 문의 처리에서 벗어나 복잡한 케이스 해결과 서비스 개선 기획에 집중할 수 있게 되었습니다. 빠른 문제 해결로 구독 해지율도 12%에서 8%로 감소했습니다.

배운 점

1. 데이터 분석이 디자인의 출발점
5,000건 이상의 CS 데이터를 분석하지 않았다면, 어떤 기능을 우선적으로 자동화해야 할지 알 수 없었을 것입니다. 사용자의 실제 pain point는 가정이 아닌 데이터에서 나옵니다.

2. 자동화와 인간적 터치의 균형
모든 것을 자동화하려 하기보다, 어떤 부분은 사람이 직접 응대하는 것이 더 나은지 구분하는 것이 중요합니다. 단순 반복 문의는 자동화하고, 감정적인 불만이나 복잡한 문제는 운영팀에 연결하는 하이브리드 접근이 가장 효과적이었습니다.

3. 지속적인 콘텐츠 관리가 핵심
아무리 좋은 디자인도 FAQ 콘텐츠가 부실하면 무용지물입니다. 출시 후 검색어 로그와 피드백을 모니터링하며 매주 FAQ를 업데이트했습니다. 디자이너는 시스템을 만들지만, 운영팀이 콘텐츠를 지속적으로 관리해야 진정한 가치가 발생합니다.

4. 에러 상태를 간과하지 말 것
초기 디자인에서는 "해피 패스"에만 집중했지만, 실제 서비스에서는 수많은 예외 상황이 발생합니다. 환불이 불가능한 경우, 결제 수단 변경이 실패하는 경우 등 10가지 이상의 엣지 케이스를 모두 디자인했습니다. 에러 메시지는 "무엇이 잘못되었는지"뿐만 아니라 "어떻게 해결할 수 있는지"까지 안내해야 사용자가 좌절하지 않습니다.
